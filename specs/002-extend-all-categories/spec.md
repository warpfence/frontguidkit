# Feature Specification: 전체 카테고리 컴포넌트 콘텐츠 확장

**Feature Branch**: `002-extend-all-categories`
**Created**: 2026-03-09
**Status**: Draft
**Input**: User description: "레이아웃 외 5개 카테고리(피드백/알림, 탐색/이동, 콘텐츠 표시, 입력/인터랙션, 데이터 시각화) 28개 컴포넌트의 와이어프레임, 용어, AI 프롬프트 콘텐츠 추가 개발. shadcn/ui 컴포넌트를 적극 활용."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 피드백/알림 카테고리 가이드 탐색 (Priority: P1)

프론트엔드 입문자가 사이드바에서 "피드백/알림" 카테고리를 펼치고, Modal/Dialog, Toast, Alert, Badge, Progress Bar 중 하나를 선택하면, 해당 컴포넌트의 와이어프레임 미리보기, 핵심 용어 4개, AI 프롬프트를 확인하고 프롬프트를 복사할 수 있다.

**Why this priority**: 피드백/알림은 모든 웹 애플리케이션에서 필수적으로 사용되는 패턴이며, 사용자 인터랙션의 핵심 요소이다. AI 코딩 시 가장 자주 요청되는 컴포넌트군이다.

**Independent Test**: 피드백/알림 카테고리의 5개 컴포넌트 중 아무 항목이나 클릭하여 와이어프레임, 용어 카드, 프롬프트가 모두 표시되고 복사 기능이 동작하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 사이드바에서 "피드백/알림" 카테고리를 클릭한 상태, **When** "Modal / Dialog"를 선택하면, **Then** 오버레이 구조를 보여주는 와이어프레임, 4개의 관련 용어 카드, AI 프롬프트가 메인 영역에 표시된다.
2. **Given** Toast/Snackbar 가이드 페이지가 열린 상태, **When** 프롬프트 복사 버튼을 클릭하면, **Then** 프롬프트 전체가 클립보드에 복사되고 "복사됨" 상태가 표시된다.
3. **Given** Badge 가이드 페이지가 열린 상태, **When** 용어 카드 섹션을 확인하면, **Then** 영어 명칭과 한국어 설명이 포함된 4개의 용어 카드가 그리드 형태로 표시된다.

---

### User Story 2 - 탐색/이동 및 콘텐츠 표시 카테고리 가이드 탐색 (Priority: P1)

AI 코딩 툴 사용자가 Breadcrumb, Tabs, Pagination, Card, Accordion 등의 컴포넌트를 사이드바에서 선택하면, 각 컴포넌트가 페이지 내 어디에 위치하는지 와이어프레임으로 확인하고, 용어를 학습하며, AI 프롬프트를 복사하여 Claude Code 등에서 바로 활용할 수 있다.

**Why this priority**: 탐색과 콘텐츠 표시 컴포넌트는 페이지 구조를 이해하는 데 필수적이며, 총 12개 컴포넌트로 전체의 약 37%를 차지한다.

**Independent Test**: 탐색/이동 카테고리의 5개 + 콘텐츠 표시 카테고리의 7개 컴포넌트 각각에서 가이드 콘텐츠가 정상 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 사이드바에서 "탐색/이동" 카테고리를 펼친 상태, **When** "Tabs"를 선택하면, **Then** 콘텐츠 영역 상단에 탭이 위치한 와이어프레임과 함께 4개 용어, 프롬프트가 표시된다.
2. **Given** 사용자가 "콘텐츠 표시" 카테고리에서 "Card"를 선택한 상태, **When** 와이어프레임을 확인하면, **Then** 카드가 그리드 내에 위치하는 구조와 내부 요소(이미지, 제목, 설명, 버튼)가 시각적으로 표시된다.
3. **Given** "Carousel / Slider" 가이드 페이지가 열린 상태, **When** 프롬프트를 복사하면, **Then** 슬라이드 이동, 인디케이터 등의 구현 요청이 포함된 프롬프트가 복사된다.

---

### User Story 3 - 입력/인터랙션 카테고리 가이드 탐색 (Priority: P2)

부트캠프 수강생이 Form/Input, Search Bar, Toggle, Date Picker 등 사용자 입력 관련 컴포넌트의 구조와 용어를 학습하고, AI 프롬프트를 활용해 각 컴포넌트를 빠르게 구현할 수 있다.

**Why this priority**: 입력 컴포넌트는 폼 기반 서비스 구현에 필수적이나, 레이아웃/피드백/탐색보다 활용 빈도가 낮다.

**Independent Test**: 입력/인터랙션 카테고리의 7개 컴포넌트 각각에서 가이드 콘텐츠가 정상 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 "입력/인터랙션" 카테고리에서 "Form / Input"을 선택한 상태, **When** 와이어프레임을 확인하면, **Then** 텍스트 입력, 선택 필드, 오류 상태 등의 구조가 표시된다.
2. **Given** "Date Picker / Calendar" 가이드 페이지가 열린 상태, **When** 용어 카드를 확인하면, **Then** 달력 팝업, 날짜 범위 등 관련 용어 4개가 표시된다.

---

### User Story 4 - 데이터 시각화 카테고리 가이드 탐색 (Priority: P2)

비개발 직군(기획자, 디자이너)이 Table, Stat Card, Chart, Heatmap 등의 데이터 시각화 컴포넌트 구조와 용어를 이해하고, 개발팀과 동일한 어휘로 소통할 수 있다.

**Why this priority**: 데이터 시각화는 대시보드/관리 도구에서 중요하지만, 일반 웹사이트 기준으로는 활용 빈도가 상대적으로 낮다.

**Independent Test**: 데이터 시각화 카테고리의 4개 컴포넌트 각각에서 가이드 콘텐츠가 정상 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 "데이터 시각화" 카테고리에서 "Table / Data Grid"를 선택한 상태, **When** 와이어프레임을 확인하면, **Then** Toolbar, Header, Row, Pagination을 포함한 테이블 구조가 표시된다.
2. **Given** "Chart (Line / Bar / Pie)" 가이드 페이지가 열린 상태, **When** 프롬프트를 복사하면, **Then** 축, 범례, 데이터 포인트 등의 구현 요청이 포함된 프롬프트가 복사된다.

---

### User Story 5 - 카테고리별 와이어프레임 차별화 (Priority: P1)

사용자가 각 카테고리의 컴포넌트를 선택했을 때, 해당 컴포넌트의 특성에 맞는 와이어프레임이 표시된다. 레이아웃 카테고리는 전체 페이지 구조를, 피드백/알림은 오버레이/인라인 표시를, 콘텐츠 표시는 그리드/리스트 구조를 각각 다른 형태로 보여준다.

**Why this priority**: 와이어프레임이 모든 컴포넌트에 동일한 레이아웃 구조로만 표시되면, 레이아웃 외 카테고리에서는 위치 맥락이 부정확해진다. 카테고리별 특성에 맞는 와이어프레임이 필수적이다.

**Independent Test**: 서로 다른 카테고리(예: 피드백의 Modal vs 탐색의 Tabs vs 입력의 Form)의 와이어프레임을 비교하여 각각 다른 구조가 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 "Modal / Dialog"를 선택한 상태, **When** 와이어프레임을 확인하면, **Then** 페이지 위에 오버레이되는 팝업 구조(Backdrop, Header, Body, Footer)가 표시된다.
2. **Given** 사용자가 "Table / Data Grid"를 선택한 상태, **When** 와이어프레임을 확인하면, **Then** 데이터 테이블 구조(Toolbar, Header Row, Data Rows, Pagination)가 표시된다.
3. **Given** 사용자가 "Accordion / Collapse"를 선택한 상태, **When** 와이어프레임을 확인하면, **Then** 접었다 펼치는 리스트 구조가 표시된다.

---

### Edge Cases

- 28개 컴포넌트 중 일부가 아직 `isReady=false` 상태일 때, 해당 컴포넌트는 "콘텐츠 준비 중" 플레이스홀더를 표시한다.
- 브라우저 창 크기가 작아 사이드바가 접힌 상태에서도 카테고리 전환이 정상 동작한다.
- 클립보드 API를 지원하지 않는 환경에서도 프롬프트 복사 대체 수단이 작동한다.
- 와이어프레임 렌더링 시 극단적으로 긴 innerElement 라벨이 레이아웃을 깨뜨리지 않는다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 5개 추가 카테고리(피드백/알림, 탐색/이동, 콘텐츠 표시, 입력/인터랙션, 데이터 시각화)에 속한 총 28개 컴포넌트의 가이드 콘텐츠를 제공해야 한다.
- **FR-002**: 각 컴포넌트는 해당 컴포넌트의 구조를 시각적으로 보여주는 와이어프레임 미리보기를 포함해야 한다.
- **FR-003**: 각 컴포넌트는 관련 핵심 용어 4개를 영어 명칭과 한국어 설명 형태로 제공해야 한다.
- **FR-004**: 각 컴포넌트는 AI 에이전트에게 구현을 요청할 수 있는 프롬프트 템플릿을 제공해야 한다.
- **FR-005**: 와이어프레임은 레이아웃 카테고리의 "전체 페이지 내 위치" 방식과 달리, 각 카테고리 특성에 맞는 독립적인 구조 다이어그램 형태로 표시되어야 한다.
- **FR-006**: 피드백/알림 카테고리의 와이어프레임은 오버레이, 인라인 배너, 토스트 위치 등 해당 컴포넌트의 출현 맥락을 시각화해야 한다.
- **FR-007**: 탐색/이동 카테고리의 와이어프레임은 페이지 내 탐색 요소의 위치와 구조를 시각화해야 한다.
- **FR-008**: 콘텐츠 표시 카테고리의 와이어프레임은 정보 블록의 내부 구성과 배치를 시각화해야 한다.
- **FR-009**: 입력/인터랙션 카테고리의 와이어프레임은 입력 필드의 상태(기본, 포커스, 오류)와 구조를 시각화해야 한다.
- **FR-010**: 데이터 시각화 카테고리의 와이어프레임은 데이터 표현 구조(테이블 행/열, 차트 축/범례 등)를 시각화해야 한다.
- **FR-011**: 프롬프트 템플릿은 구조(레이아웃/배치)와 동작(인터랙션/상태 변화)을 포함해야 한다.
- **FR-012**: 사이드바에서 새로 추가된 컴포넌트를 선택하면 기존 레이아웃 카테고리와 동일한 방식으로 페이지가 전환되어야 한다.
- **FR-013**: 모든 컴포넌트의 `isReady` 플래그가 `true`로 설정되어 플레이스홀더 대신 실제 콘텐츠가 표시되어야 한다.
- **FR-014**: 와이어프레임 컴포넌트는 카테고리별 특성에 맞게 다양한 레이아웃 템플릿을 지원해야 한다.

### Key Entities

- **Component**: 각 UI 컴포넌트를 나타내는 데이터 단위. slug(고유 식별자), name, description, categoryId, isReady 상태, wireframe 데이터, terms 배열(4개), prompt 문자열을 포함한다.
- **WireframeData**: 컴포넌트의 시각적 구조를 정의하는 데이터. 강조 영역(highlightArea)과 내부 요소 목록(innerElements)으로 구성된다. 카테고리별로 다른 레이아웃 템플릿을 참조할 수 있다.
- **Term**: 용어 정의 단위. 영어 명칭(en)과 한국어 설명(ko)으로 구성된다.
- **Category**: 컴포넌트를 분류하는 상위 그룹. 6개 고정 카테고리로 구성되며, 이미 정의되어 있다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사이드바에서 6개 카테고리의 모든 33개 컴포넌트를 선택할 수 있고, 각각 와이어프레임·용어·프롬프트가 표시된다.
- **SC-002**: 28개 신규 컴포넌트 각각의 프롬프트를 1회 클릭으로 복사할 수 있다.
- **SC-003**: 각 컴포넌트의 용어 카드가 정확히 4개씩 표시된다.
- **SC-004**: 서로 다른 카테고리의 컴포넌트를 선택했을 때 각각 해당 카테고리 특성에 맞는 와이어프레임이 표시된다 (최소 3가지 이상의 와이어프레임 레이아웃 변형).
- **SC-005**: 페이지 전환 시 콘텐츠가 즉시 표시되며, "콘텐츠 준비 중" 플레이스홀더가 나타나지 않는다.

## Assumptions

- 기존 데이터 구조(Component, WireframeData, Term 타입)를 유지하되, 와이어프레임의 highlightArea와 innerElements 패턴을 카테고리별로 확장한다.
- 와이어프레임 미리보기 컴포넌트(`wireframe-preview.tsx`)는 기존 5개 영역(header, hero, navbar, body, footer) 외에 새로운 레이아웃 템플릿을 지원하도록 확장한다.
- shadcn/ui 컴포넌트를 적극 활용하여 UI 일관성을 유지한다.
- 프롬프트 템플릿은 한국어로 작성하며, AI 에이전트(Claude Code, Codex, GeminiCLI)에서 범용적으로 사용 가능한 형태로 제공한다.
- 컴포넌트 데이터는 정적 TypeScript 객체로 관리하며, 별도 데이터베이스나 API는 사용하지 않는다.
