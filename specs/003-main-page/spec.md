# Feature Specification: FrontGuidKit 메인페이지

**Feature Branch**: `003-main-page`
**Created**: 2026-03-09
**Status**: Draft
**Input**: project-proposal.md — 메인페이지 구현 (Home 네비게이션, Hero 섹션, 카테고리 카드 그리드, 다크/라이트 모드, Footer, 디자인 개선)

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 메인페이지 카테고리 탐색 (Priority: P1)

사용자가 FrontGuidKit에 처음 접속하면 메인페이지가 표시된다. 페이지 상단에 프로젝트 제목(`</> FrontGuidKit`)과 부제목(`프론트엔드 개발 가이드 & AI 프롬프트`), 간단한 설명이 보인다. 아래로 스크롤하면 6개 카테고리(레이아웃, 피드백/알림, 탐색/이동, 콘텐츠 표시, 입력/인터랙션, 데이터 시각화)가 섹션 타이틀로 구분되어 있고, 각 카테고리 아래에 해당 컴포넌트들이 3열 카드 형태로 나열된다. 카드를 클릭하면 해당 컴포넌트의 상세 가이드 페이지로 이동한다.

**Why this priority**: 메인페이지의 핵심 목적인 전체 컴포넌트 탐색 허브 기능. 이것만으로도 사용자에게 가치를 제공한다.

**Independent Test**: 메인페이지(`/`)에 접속하여 Hero 섹션과 6개 카테고리 섹션이 표시되는지, 카드 클릭 시 상세 페이지로 이동하는지 확인

**Acceptance Scenarios**:

1. **Given** 사용자가 사이트에 접속, **When** `/` URL로 이동, **Then** Hero 섹션에 제목(`</> FrontGuidKit`), 부제목(`프론트엔드 개발 가이드 & AI 프롬프트`), 설명 텍스트가 표시된다
2. **Given** 메인페이지가 표시된 상태, **When** 아래로 스크롤, **Then** 6개 카테고리가 섹션 타이틀과 함께 순서대로 표시된다
3. **Given** 카테고리 섹션이 보이는 상태, **When** 각 카테고리 확인, **Then** 해당 카테고리의 컴포넌트들이 3열 카드 그리드로 표시된다
4. **Given** 컴포넌트 카드가 표시된 상태, **When** 특정 카드를 클릭, **Then** `/components/[slug]` 상세 페이지로 이동한다

---

### User Story 2 - Home 네비게이션 및 사이드바 상태 (Priority: P1)

사용자가 컴포넌트 상세 페이지를 보고 있다가 메인페이지로 돌아가고 싶을 때, 사이드바 상단의 Home 아이콘을 클릭하면 메인페이지로 이동한다. 메인페이지에 도착하면 사이드바의 모든 카테고리 Collapsible이 접힌 상태로 표시된다.

**Why this priority**: 메인페이지로의 복귀 경로가 없으면 메인페이지 자체의 가치가 반감된다. P1으로 메인페이지와 동일 우선순위.

**Independent Test**: 아무 상세 페이지에서 Home 아이콘을 클릭하여 메인페이지로 이동되는지, Collapsible이 모두 접히는지 확인

**Acceptance Scenarios**:

1. **Given** 사이드바가 표시된 상태, **When** 사이드바 확인, **Then** 상단에 Home 아이콘이 표시된다
2. **Given** 컴포넌트 상세 페이지(`/components/header`)에 있는 상태, **When** Home 아이콘을 클릭, **Then** 메인페이지(`/`)로 이동한다
3. **Given** Home 아이콘을 클릭하여 메인페이지로 이동, **When** 사이드바 확인, **Then** 모든 카테고리 Collapsible이 접힌 상태이다

---

### User Story 3 - 다크/라이트 모드 전환 (Priority: P2)

사용자가 사이트에 접속하면 운영체제의 시스템 테마 설정에 맞는 모드(다크 또는 라이트)로 자동 표시된다. 사용자가 테마를 변경하고 싶을 때, 헤더 우측의 테마 토글 아이콘(Sun/Moon)을 클릭하면 즉시 반대 모드로 전환된다.

**Why this priority**: 시각적 경험을 개선하지만 핵심 기능(탐색)과는 독립적이므로 P2.

**Independent Test**: 시스템 다크모드에서 접속 시 다크 테마 적용 확인, 토글 버튼 클릭 시 라이트 모드 전환 확인

**Acceptance Scenarios**:

1. **Given** 시스템이 다크모드인 상태, **When** 사이트에 처음 접속, **Then** 다크 테마로 표시된다
2. **Given** 시스템이 라이트모드인 상태, **When** 사이트에 처음 접속, **Then** 라이트 테마로 표시된다
3. **Given** 다크모드로 표시된 상태, **When** 헤더 우측의 테마 토글 아이콘을 클릭, **Then** 라이트모드로 즉시 전환된다
4. **Given** 라이트모드로 표시된 상태, **When** 테마 토글 아이콘을 클릭, **Then** 다크모드로 즉시 전환된다

---

### User Story 4 - Footer 정보 확인 (Priority: P3)

사용자가 페이지를 끝까지 스크롤하면 Footer가 표시된다. Footer에는 3열 구조로 링크 모음, 저작권 표기, 소셜 아이콘이 배치되어 프로젝트 정보와 관련 링크를 확인할 수 있다.

**Why this priority**: 보조 정보 영역으로 핵심 기능에 영향을 주지 않으므로 P3.

**Independent Test**: 페이지 하단에 Footer가 3열로 렌더링되는지 확인

**Acceptance Scenarios**:

1. **Given** 메인페이지가 표시된 상태, **When** 페이지 하단으로 스크롤, **Then** Footer가 3열 구조로 표시된다
2. **Given** Footer가 표시된 상태, **When** 내용 확인, **Then** 첫째 열에 링크 모음, 둘째 열에 저작권 표기, 셋째 열에 소셜 아이콘이 있다
3. **Given** 컴포넌트 상세 페이지에 있는 상태, **When** 페이지 하단으로 스크롤, **Then** 동일한 Footer가 표시된다

---

### User Story 5 - 디자인 개선 (Priority: P2)

사이트 전체의 가독성을 높이기 위해 헤더의 폭이 20% 확대되고, 헤더 내 프로젝트명과 부제목 폰트 크기도 비례하여 커진다. 모든 페이지의 전체 폰트 크기가 20% 증가하여 텍스트를 더 쉽게 읽을 수 있다.

**Why this priority**: 사용성 개선이지만 기능 동작에는 영향 없으므로 P2.

**Independent Test**: 헤더 폭과 폰트 크기가 기존 대비 20% 증가했는지 시각적으로 확인, 기존 컴포넌트 상세 페이지 레이아웃이 깨지지 않는지 확인

**Acceptance Scenarios**:

1. **Given** 사이트에 접속, **When** 헤더를 확인, **Then** 헤더 높이가 기존보다 약 20% 크고 프로젝트명/부제목 폰트도 커져 있다
2. **Given** 사이트에 접속, **When** 본문 텍스트를 확인, **Then** 전체 폰트 크기가 기존 대비 약 20% 증가되어 있다
3. **Given** 폰트 크기가 증가된 상태, **When** 컴포넌트 상세 페이지를 확인, **Then** 레이아웃이 깨지지 않고 정상적으로 표시된다

---

### Edge Cases

- 카테고리에 컴포넌트가 0개인 경우: 해당 카테고리 섹션은 표시하지 않는다
- 매우 긴 컴포넌트 이름의 카드: 카드 내에서 텍스트가 잘리거나 줄바꿈 처리된다
- 모바일 환경에서 3열 카드 그리드: 화면 폭에 따라 1열 또는 2열로 자동 조정된다
- 사이드바가 접힌 상태에서 Home 아이콘: 아이콘만 표시되더라도 클릭 시 정상적으로 메인페이지로 이동
- 테마 토글 후 페이지 이동: 선택한 테마가 페이지 전환 후에도 유지된다

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 루트 경로(`/`)에서 메인페이지를 표시해야 한다 (기존 redirect 제거)
- **FR-002**: 메인페이지 Body 상단에 Hero 섹션을 표시해야 한다 — 제목(`</> FrontGuidKit`), 부제목(`프론트엔드 개발 가이드 & AI 프롬프트`), 간단한 설명
- **FR-003**: Hero 섹션 아래에 6개 카테고리를 섹션 타이틀(아이콘 포함)과 함께 순서대로 표시해야 한다
- **FR-004**: 각 카테고리 섹션 아래에 해당 컴포넌트들을 3열 카드 그리드로 표시해야 한다
- **FR-005**: 카드에는 컴포넌트 이름과 와이어프레임 미리보기 영역(placeholder)을 포함해야 한다
- **FR-006**: 카드 클릭 시 해당 컴포넌트의 상세 페이지(`/components/[slug]`)로 이동해야 한다
- **FR-007**: 사이드바 상단에 Home 아이콘을 추가하고, 클릭 시 메인페이지(`/`)로 이동해야 한다
- **FR-008**: 메인페이지(`/`)에 진입하면 사이드바의 모든 카테고리 Collapsible이 접힌 상태여야 한다
- **FR-009**: 사이트 접속 시 운영체제의 시스템 테마 설정에 맞는 모드(다크/라이트)를 자동 감지하여 적용해야 한다
- **FR-010**: 헤더 우측에 테마 토글 아이콘(Sun/Moon)을 배치하고, 클릭 시 다크↔라이트 모드를 전환해야 한다
- **FR-011**: 선택한 테마는 페이지 이동 후에도 유지되어야 한다
- **FR-012**: 페이지 하단에 3열 구조 Footer를 표시해야 한다 — 1열: 링크 모음, 2열: 저작권 표기, 3열: 소셜 아이콘
- **FR-013**: Footer는 메인페이지뿐 아니라 모든 페이지에서 표시되어야 한다
- **FR-014**: 헤더의 높이를 기존 대비 약 20% 확대하고, 헤더 내 프로젝트명/부제목 폰트 크기도 비례하여 키워야 한다
- **FR-015**: 전체 사이트의 기본 폰트 크기를 약 20% 증가시켜야 한다
- **FR-016**: 카드 그리드는 반응형으로 모바일에서 1열, 태블릿에서 2열, 데스크탑에서 3열로 표시되어야 한다

### Key Entities

- **Category**: 컴포넌트를 그룹화하는 분류 단위. id, name, icon, order, components 속성을 가지며 메인페이지에서 섹션 타이틀로 사용된다.
- **ComponentData**: 개별 UI 컴포넌트 가이드 데이터. slug, name, categoryId 등 속성을 가지며 메인페이지에서 카드로 표시된다.
- **Theme**: 사용자의 시각적 테마 선택. 시스템 감지 또는 수동 토글로 결정되며 다크/라이트 두 가지 값을 가진다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 메인페이지(`/`)에 접속하면 6개 카테고리 섹션과 33개 컴포넌트 카드가 정상적으로 렌더링된다
- **SC-002**: 아무 컴포넌트 카드를 클릭하면 1초 이내에 해당 상세 페이지로 이동한다
- **SC-003**: Home 아이콘 클릭 시 1초 이내에 메인페이지로 이동하고 모든 Collapsible이 접힌 상태이다
- **SC-004**: 시스템 다크모드 환경에서 접속 시 다크 테마가 적용되고, 라이트모드에서는 라이트 테마가 적용된다
- **SC-005**: 테마 토글 버튼 클릭 시 즉시(500ms 이내) 테마가 전환되고, 페이지 이동 후에도 선택 테마가 유지된다
- **SC-006**: Footer가 모든 페이지에서 3열 구조로 정상 표시된다
- **SC-007**: 빌드와 린트가 에러 없이 통과한다
- **SC-008**: 모바일(≤640px), 태블릿(641~1024px), 데스크탑(≥1025px) 해상도에서 카드 그리드가 각각 1열, 2열, 3열로 정상 표시된다

## Assumptions

- 카드의 와이어프레임 미리보기 이미지는 현재 placeholder(아이콘 또는 빈 영역)로 대체하며, 추후 실제 이미지로 교체할 수 있는 구조로 구현한다.
- Footer의 링크 모음, 소셜 아이콘의 구체적인 URL은 임시 값(#)을 사용하며 추후 실제 URL로 교체한다.
- 폰트 크기 20% 증가는 HTML root의 font-size를 조정하여 rem 기반으로 전체 적용한다.
- 기존 컴포넌트 상세 페이지의 레이아웃은 폰트 크기 변경 후에도 정상 동작해야 한다.
